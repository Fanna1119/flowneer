# v0.9.6

## `PluginContext` — typed `this` for plugin methods

Plugin methods no longer need `(this as any)._setHooks(...)`. A new exported type `PluginContext` provides a fully-typed `this` that includes `_setHooks`, eliminating all `as any` casts from plugin authoring.

### What changed

- **New type `PluginContext`** exported from `flowneer` / `../../Flowneer`:
  ```typescript
  type PluginContext = FlowBuilder<any, any> & {
    _setHooks(
      hooks: Partial<FlowHooks<any, any>>,
      filter?: StepFilter,
    ): () => void;
  };
  ```
- **`FlowneerPlugin`** updated — the `this` parameter of every plugin method is now `PluginContext` instead of `FlowBuilder<any, any>`.
- **All built-in plugins** updated to use `this: PluginContext` and call `this._setHooks(...)` directly.

### Migrating your own plugins

Before:

```typescript
import type { FlowBuilder, FlowneerPlugin } from "flowneer";

export const withMyPlugin: FlowneerPlugin = {
  withMyPlugin(this: FlowBuilder<any, any>) {
    (this as any)._setHooks({
      afterStep: () => {
        /* ... */
      },
    });
    return this;
  },
};
```

After:

```typescript
import type { FlowneerPlugin, PluginContext } from "flowneer";

export const withMyPlugin: FlowneerPlugin = {
  withMyPlugin(this: PluginContext) {
    this._setHooks({
      afterStep: () => {
        /* ... */
      },
    });
    return this;
  },
};
```

If you do not add an explicit `this:` annotation, `PluginContext` is inferred automatically from `FlowneerPlugin` — the cast is gone either way.

## `flow.addHooks()` — public runtime hook registration

`addHooks` is now a first-class public method on `FlowBuilder` (and `CoreFlowBuilder`). It was previously documented but unimplemented.

### Usage

```typescript
const dispose = flow.addHooks(
  { beforeStep: (meta) => console.log("->", meta.label) },
  ["llm:*"],
);

await flow.run(shared);

dispose(); // removes the hooks
```

### Signature

```typescript
flow.addHooks(hooks: Partial<FlowHooks>, filter?: StepFilter): () => void
```

Accepts the same `hooks` and optional `StepFilter` as `_setHooks`, and returns a `dispose()` function to remove the registered hooks. Unlike `_setHooks` (which is `protected` and intended for plugin internals), `addHooks` is public API for consumers.

### When to use it

- **Per-instance observability** — attach a logger or metric collector to one specific flow instance without defining a plugin.
- **Test instrumentation** — assert on hook invocations for a single test run.
- **Request-scoped hooks** — register and dispose hooks around a single `.run()` call in a server handler.
