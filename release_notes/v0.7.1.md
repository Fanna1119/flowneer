# v0.7.1

## Bug fixes

### `stream()` no longer accumulates hooks on repeated calls

Calling `.stream()` multiple times on the same flow instance previously caused `beforeStep` and `afterStep` hooks to grow permanently with each call. On the second call events would fire twice; on the third, three times. The hooks are now cleaned up in a `finally` block when the generator returns or is abandoned, and the previous `__stream` handler is restored. Streaming a flow in a loop is now safe.

### `withReActLoop` ‚Äî iteration counter resets between runs

The iteration guard was stored in a closure variable (`let iterations = 0`) captured at flow-construction time. After the first `.run()` exhausted the budget, the counter was stuck at `maxIterations` and the loop condition was immediately false on every subsequent call ‚Äî the agent silently did nothing after its first run.

The counter is now stored on `shared.__reactIterations` and is cleaned up at the end of each run alongside the other internal `__react*` flags. Flows using `.withReActLoop()` or `createAgent()` are fully reusable and can be `.run()` as many times as needed.

### `withFallback` and `withTryCatch` ‚Äî `InterruptError` now propagates correctly

`InterruptError` is Flowneer's mechanism for explicit flow interruption (used by `withHumanNode` and callable from any step). It is supposed to escape all hooks and propagate to the caller unchanged. However, both `withFallback` and `withTryCatch` wrap steps at the `wrapStep` layer ‚Äî below the guard in the core executor ‚Äî meaning an `InterruptError` thrown inside a wrapped step was being caught and treated as a normal error.

Both plugins now re-throw `InterruptError` immediately before any fallback or recovery logic runs.

### `parallel()` ‚Äî `AbortSignal` is now checked per branch

Aborting a flow mid-parallel previously waited for all concurrently running functions to settle before the abort took effect. Each parallel function now checks `signal.throwIfAborted()` on entry, so an in-flight abort is respected as soon as each running branch completes.

### `_setHooks` dispose is now order-independent

The dispose function returned by `addHooks()` / `_setHooks()` previously captured the array index at registration time. Removing hooks out of LIFO order (e.g. two plugins disposing in reverse sequence) would shift later entries and silently remove the wrong hook. The dispose function now resolves the index at call time via `indexOf`, which is always correct regardless of removal order.

## New features

### Step labels

All step builder methods now accept a `label` string. Labels appear in:

- `StepMeta.label` ‚Äî visible to all hook callbacks (`beforeStep`, `afterStep`, `onError`, `wrapStep`)
- Error messages ‚Äî `FlowError: "fetchUser" step 2 ‚Ä¶` instead of `FlowError: step 2 ‚Ä¶`

```typescript
flow
  .then(fetchUser, { label: "fetchUser" })
  .branch(
    routeByIntent,
    { buy: handleBuy, refund: handleRefund },
    { label: "intentRouter" },
  )
  .loop(
    (s) => !s.done,
    (b) => b.then(poll),
    { label: "pollLoop" },
  )
  .batch(
    (s) => s.items,
    (b) => b.then(process),
    { label: "itemBatch" },
  )
  .parallel([workerA, workerB], { label: "parallelFetch" });
```

`NodeOptions` now includes:

```typescript
interface NodeOptions<S, P> {
  retries?: NumberOrFn<S, P>;
  delaySec?: NumberOrFn<S, P>;
  timeoutMs?: NumberOrFn<S, P>;
  label?: string; // new
}
```

## Architecture refactor ‚Äî plugin-first `CoreFlowBuilder`

The execution engine has been split into a minimal base class and a set of independently registered step-type handlers. This is an internal restructuring ‚Äî all existing `FlowBuilder` APIs are unchanged.

### New: `CoreFlowBuilder`

`CoreFlowBuilder` is now exported as a first-class public API. It contains only the orchestration engine: the hook registry, a static step-type dispatch table, and `run()` / `stream()` / `_execute()`. It ships with **no built-in step types** ‚Äî those are registered as plugins. The reason for this inversion is to allow users to build custom flow-like systems on top of the core execution engine without inheriting any built-in step types or their assumptions.

Use `CoreFlowBuilder` as a base when you want a completely blank-slate builder:

```typescript
import { CoreFlowBuilder } from "flowneer";

CoreFlowBuilder.registerStepType("myStep", async (step, ctx) => {
  await doWork(ctx.shared);
  return undefined; // or "#anchorName" to goto
});
```

`FlowBuilder` continues to extend `CoreFlowBuilder` and registers all primitive step types on import ‚Äî no consumer changes required.

### New: `StepContext` and `StepHandler` types

Two new types are exported for authoring custom step-type handlers:

```typescript
export interface StepContext<S, P> {
  shared: S;
  params: P;
  signal?: AbortSignal;
  hooks: ResolvedHooks<S, P>;
  meta: StepMeta;
  builder: CoreFlowBuilder<S, P>;
}

export type StepHandler = (
  step: any,
  ctx: StepContext<any, any>,
) => Promise<string | undefined>;
```

### Step handlers extracted to `src/core/steps/`

Each built-in step type is now a standalone module that exports a named handler function:

```
src/core/steps/
  fn.ts       ‚Äî fnHandler
  branch.ts   ‚Äî branchHandler
  loop.ts     ‚Äî loopHandler
  batch.ts    ‚Äî batchHandler
  parallel.ts ‚Äî parallelHandler
  anchor.ts   ‚Äî anchorHandler (no-op marker)
  index.ts    ‚Äî exports registerBuiltinSteps()
```

`FlowBuilder` calls `registerBuiltinSteps()` once on module load, which registers all handlers explicitly via `CoreFlowBuilder.registerStepType()`. This replaces the previous bare side-effect import pattern (`import "./core/steps/index"`), which was incompatible with `"sideEffects": false` in `package.json` and caused bundlers (including tsup and downstream consumer bundlers) to emit warnings and potentially tree-shake the registrations away.

## Performance & internal improvements

These are all internal ‚Äî no API changes.

### Anchor map cached per-instance

`_execute()` previously scanned `steps` for anchors and rebuilt the name‚Üíindex map on every `run()` call. The map is now cached on the builder instance as `_anchorMap` and only invalidated when steps are added (via the new `_pushStep()` helper used internally by all `FlowBuilder` step methods). Flows that are `run()` in a tight loop (e.g. per-request agents) no longer pay this cost on every invocation.

### `_runStep` and `_dispatchStep` merged

`_dispatchStep` was a private method called exclusively by `_runStep`, forwarding all arguments unchanged. The two have been merged into a single method: one fewer async stack frame and one fewer closure allocation per step.

### `wrapStep` fast-path

When no `wrapStep` hooks are registered (the common case: no plugins attached), the executor previously still ran `reduceRight` over an empty array and allocated a wrapper closure per step. A guard (`hooks.wrapStep.length === 0 && timeoutMs === 0`) now calls the handler directly, skipping all allocation on every step in the unobserved hot path.

## New feature: `anchor(name, maxVisits?)``.anchor()` now accepts an optional second argument, `maxVisits`, as a built-in cycle guard. When a `"#name"` goto lands on an anchor more times than `maxVisits` allows in a single `run()`, an error is thrown immediately ‚Äî no plugin required.

```typescript
flow
  .anchor("refine", 5) // allow up to 5 gotos back to this anchor
  .then(generateDraft)
  .then((s) => (s.score < 0.9 ? "#refine" : undefined))
  .then(publish);
```

The visit counter is per-`run()` and resets automatically on each call.

The `AnchorStep` type gains the corresponding optional field:

```typescript
interface AnchorStep {
  type: "anchor";
  name: string;
  maxVisits?: number; // new
}
```

## Deprecation: `withCycles`

`withCycles` is now deprecated in favour of the built-in `anchor(name, maxVisits)` parameter above. It will be removed in a future major version.

```typescript
// before
flow
  .withCycles(5)
  .anchor("refine")
  .then(step)
  .then((s) => "#refine");

// after
flow
  .anchor("refine", 5)
  .then(step)
  .then((s) => "#refine");
```
