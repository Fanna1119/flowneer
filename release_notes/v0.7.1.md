# v0.7.1

## Bug fixes

### `stream()` no longer accumulates hooks on repeated calls

Calling `.stream()` multiple times on the same flow instance previously caused `beforeStep` and `afterStep` hooks to grow permanently with each call. On the second call events would fire twice; on the third, three times. The hooks are now cleaned up in a `finally` block when the generator returns or is abandoned, and the previous `__stream` handler is restored. Streaming a flow in a loop is now safe.

### `withReActLoop` — iteration counter resets between runs

The iteration guard was stored in a closure variable (`let iterations = 0`) captured at flow-construction time. After the first `.run()` exhausted the budget, the counter was stuck at `maxIterations` and the loop condition was immediately false on every subsequent call — the agent silently did nothing after its first run.

The counter is now stored on `shared.__reactIterations` and is cleaned up at the end of each run alongside the other internal `__react*` flags. Flows using `.withReActLoop()` or `createAgent()` are fully reusable and can be `.run()` as many times as needed.

### `withFallback` and `withTryCatch` — `InterruptError` now propagates correctly

`InterruptError` is Flowneer's mechanism for explicit flow interruption (used by `withHumanNode` and callable from any step). It is supposed to escape all hooks and propagate to the caller unchanged. However, both `withFallback` and `withTryCatch` wrap steps at the `wrapStep` layer — below the guard in the core executor — meaning an `InterruptError` thrown inside a wrapped step was being caught and treated as a normal error.

Both plugins now re-throw `InterruptError` immediately before any fallback or recovery logic runs.

### `parallel()` — `AbortSignal` is now checked per branch

Aborting a flow mid-parallel previously waited for all concurrently running functions to settle before the abort took effect. Each parallel function now checks `signal.throwIfAborted()` on entry, so an in-flight abort is respected as soon as each running branch completes.

## New features

### Step labels

All step builder methods now accept a `label` string. Labels appear in:

- `StepMeta.label` — visible to all hook callbacks (`beforeStep`, `afterStep`, `onError`, `wrapStep`)
- Error messages — `FlowError: "fetchUser" step 2 …` instead of `FlowError: step 2 …`

```typescript
flow
  .then(fetchUser, { label: "fetchUser" })
  .branch(
    routeByIntent,
    { buy: handleBuy, refund: handleRefund },
    { label: "intentRouter" },
  )
  .loop(
    (s) => !s.done,
    (b) => b.then(poll),
    { label: "pollLoop" },
  )
  .batch(
    (s) => s.items,
    (b) => b.then(process),
    { label: "itemBatch" },
  )
  .parallel([workerA, workerB], { label: "parallelFetch" });
```

`NodeOptions` now includes:

```typescript
interface NodeOptions<S, P> {
  retries?: NumberOrFn<S, P>;
  delaySec?: NumberOrFn<S, P>;
  timeoutMs?: NumberOrFn<S, P>;
  label?: string; // new
}
```
