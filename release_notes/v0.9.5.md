# v0.9.5

## `StepFilter` negation support

String-array filters now support a `!` prefix to **exclude** specific steps. This works alongside existing exact-label and glob-wildcard patterns, and with all hooks that accept a `StepFilter` (`_setHooks`, `addHooks`, and every plugin that exposes a `filter` parameter such as `withPerfAnalyzer`, `withManualStepping`, `withRateLimit`, etc.).

### Negation veto semantics

1. Any entry starting with `!` is a negation. The rest of the string is treated as a normal pattern (exact or glob).
2. If **any negation matches** the step's label, the hook does not fire — regardless of whether a positive pattern also matches.
3. If **positive patterns are present**, the step must still match at least one.
4. A **negation-only array** matches every step that is not explicitly excluded.

### Examples

```typescript
// Apply everywhere except human-in-loop steps
flow.withRateLimit({ intervalMs: 500 }, ["!human:*"]);

// Apply to llm:* steps only, but never human:* steps
flow.withPerfAnalyzer({}, ["!human:*", "llm:*"]);

// Negation veto beats a matching wildcard in the same array
// → fires on all llm:* steps except llm:generate
flow.addHooks({ beforeStep: log }, ["!llm:generate", "llm:*"]);

// Exact negation — skip one specific step, fire on everything else
flow.addHooks({ beforeStep: audit }, ["!db:write"]);
```

### Behaviour reference

| Filter array                | `human:review` | `llm:generate` | `db:save` | unlabelled |
| --------------------------- | -------------- | -------------- | --------- | ---------- |
| `["llm:*"]`                 | ✗              | ✓              | ✗         | ✗          |
| `["!human:*"]`              | ✗              | ✓              | ✓         | ✓          |
| `["!human:*", "llm:*"]`     | ✗              | ✓              | ✗         | ✗          |
| `["!llm:generate","llm:*"]` | ✗              | ✗              | ✗         | ✗          |

> **Unlabelled steps** cannot be vetoed by a negation (there is no label to match against). They are included by a negation-only filter but excluded by any positive-pattern filter — consistent with existing behaviour.

### No breaking changes

The change is fully backwards-compatible. Existing filters containing no `!`-prefix entries behave identically to before.

## `swarm` — decentralized multi-agent preset

A new `swarm()` factory function in `flowneer/presets/agent` implements peer-to-peer agent handoffs without a central manager. Any agent can route to any other agent at runtime by calling `handoffTo()`.

### Usage

```typescript
import { swarm, handoffTo } from "flowneer/presets/agent";
import type { SwarmAgent, SwarmState } from "flowneer/presets/agent";

interface MyState extends SwarmState {
  messages: { role: "user" | "assistant"; content: string }[];
}

const triage: SwarmAgent<MyState> = {
  name: "triage",
  description: "Routes queries to the right specialist",
  fn: async (shared) => {
    const topic = classify(shared.messages);
    handoffTo(shared, topic === "billing" ? "billing" : "support");
  },
};

const billing: SwarmAgent<MyState> = {
  name: "billing",
  description: "Handles billing and payment questions",
  fn: async (shared) => {
    shared.messages.push({
      role: "assistant",
      content: await billingLlm(shared),
    });
    // No handoffTo → turn ends here
  },
};

const flow = swarm([triage, billing, support], { defaultAgent: "triage" });
await flow.run({ messages: [{ role: "user", content: "I need a refund" }] });
```

### Handoff semantics

- `handoffTo(shared, agentName, reason?)` — called inside an agent `fn` to request a handoff.
- If the target name is not in the swarm the handoff is **silently dropped** and the turn ends.
- The loop continues until an agent finishes without calling `handoffTo`, or `maxHandoffs` is reached.

### Options

| Option          | Default | Description                                                        |
| --------------- | ------- | ------------------------------------------------------------------ |
| `defaultAgent`  | —       | **Required.** Starting agent on first `.run()`.                    |
| `maxHandoffs`   | `5`     | Max hops per `.run()` call before `onMaxHandoffs` fires.           |
| `onHandoff`     | —       | Callback after each accepted handoff `(from, to, reason, shared)`. |
| `onMaxHandoffs` | —       | Callback when the hop limit is reached.                            |

### State

`currentAgent` persists between `.run()` calls — the swarm remembers which agent last had control across multi-turn conversations. `turnCount` resets to `0` on each `.run()` call. The internal `__swarmDone` and `__swarmHandoff` fields are cleaned up before each run returns.

### Composing with plugins

Pass a `FlowBuilder.extend()` subclass as the optional third argument to `swarm()` to apply plugins to any internal swarm step. The `swarm:*` step labels make individual steps targetable via `StepFilter`:

```typescript
import { FlowBuilder } from "flowneer";
import { swarm } from "flowneer/presets/agent";
import { withTiming } from "flowneer/plugins/observability";
import { withRateLimit } from "flowneer/plugins/llm";

const AppFlow = FlowBuilder.extend([withTiming, withRateLimit]);

const flow = swarm(agents, { defaultAgent: "triage" }, AppFlow);

(flow as InstanceType<typeof AppFlow>)
  .withRateLimit({ intervalMs: 200 }) // throttle every step
  .withTiming(["!swarm:init", "!swarm:cleanup"]); // time everything except bookkeeping
```

Internal step labels exposed by `swarm()`:

| Label            | Description                                  |
| ---------------- | -------------------------------------------- |
| `swarm:init`     | Per-run initialisation                       |
| `swarm:router`   | Optional LLM router call                     |
| `swarm:loop`     | The handoff loop step                        |
| `swarm:dispatch` | Agent `fn` invocation (inside loop body)     |
| `swarm:handoff`  | Handoff check / callbacks (inside loop body) |
| `swarm:cleanup`  | Post-loop cleanup                            |

`swarm:init`, `swarm:router`, `swarm:loop`, and `swarm:cleanup` are on the outer flow and are reachable by plugin hooks. `swarm:dispatch` and `swarm:handoff` run inside the loop body's own `FlowBuilder` instance — they carry labels (visible in error messages) but are not reached by hooks registered on the outer flow.

See the full documentation at [`docs/presets/agent/swarm.md`](../docs/presets/agent/swarm.md).
